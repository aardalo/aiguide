# 🚫 BLOCKER RESOLUTION: CODE REVIEW SLA (R-005)\n\n**Decision Date**: March 2, 2026  \n**Status**: ✅ **LOCKED** - Code review process ready  \n**Owner**: ✅ Reviewer Agent\n\n---\n\n## Code Review Service Level Agreement (SLA)\n\n**Purpose**: Define clear code review expectations to prevent reviewer bottlenecks during SPRINT-001\n\n---\n\n## Review Turnaround Targets\n\n| Scenario | Max Time | Owner |\n|----------|----------|-------|\n| **Regular PR Review** | 4 hours from submission | ✅ Reviewer Agent |\n| **Critical Bug Fix** | 1 hour (if available) | ✅ Reviewer Agent |\n| **Blocker PR** (blocking other work) | 2 hours (priority) | ✅ Reviewer Agent |\n| **Re-review after changes** | 2 hours | ✅ Reviewer Agent |\n\n**Daily Review Window**: 09:00-17:00 UTC (8-hour window each day)\n\n---\n\n## Code Review Process\n\n### 1. Implementer Submits PR\n\n**PR Submission Checklist** (Implementer responsibility):\n- [ ] Code passes local `npm test` (all tests pass)\n- [ ] Code passes linting: `ruff check .` and ESLint\n- [ ] TypeScript check passes: `tsc --noEmit`\n- [ ] PR title follows format: `[EPIC-001] TASK-XXX: Brief description`\n- [ ] PR description includes:\n  - Link to task in `/opt/backlog/tracking/SPRINT-001-AGENT-ASSIGNMENTS.md`\n  - Summary of changes\n  - Any risks or notes for reviewer\n- [ ] All acceptance criteria from task marked as complete\n\n**Friday Example PR Description**:\n```\n## Task: [IMPL-002] TASK-003: CRUD API Endpoints\n\nImplements all trip REST endpoints (POST create, GET list, GET detail, PATCH update, DELETE).\n\n### Changes\n- Created `src/app/api/trips/route.ts` with POST and GET handlers\n- Created `src/app/api/trips/[id]/route.ts` with GET, PATCH, DELETE handlers\n- Created `src/lib/schemas/trip.ts` with Zod validation schemas\n- Added integration tests in `tests/integration/api/trips.test.ts`\n\n### Testing\n- [x] All endpoints tested locally with cURL\n- [x] Integration tests: 12/12 passing\n- [x] Validation errors tested and match spec\n\n### Acceptance Criteria\n- [x] All 5 endpoints working (POST, GET list, GET detail, PATCH, DELETE)\n- [x] Zod validation enforces rules\n- [x] Error responses match API contract\n- [x] Tests cover happy path + error cases\n\n### Notes\nNo breaking changes. API contract followed exactly.\n\n### Risks\nNone identified.\n```\n\n### 2. Reviewer Acknowledges PR\n\nWithin **1 hour** of PR submission:\n- [ ] Reviewer assigns self to PR\n- [ ] Reviewer posts comment: \"Reviewing now. ETA [time]\"\n\n### 3. Reviewer Performs Review\n\n**Review Dimensions** (score each 1-5):\n\n1. **Correctness (1-5)**: Does it work? Does it match the spec? Are there bugs?\n   - 5: Perfect, no issues\n   - 3: Works but has minor issues or edge cases not handled\n   - 1: Broken, doesn't meet spec\n\n2. **Readability (1-5)**: Can others understand this code?\n   - 5: Clear variable names, good comments, easy to follow\n   - 3: Mostly clear, some confusing sections\n   - 1: Unreadable, bad variable names, no comments\n\n3. **Maintainability (1-5)**: Can this be modified/updated later?\n   - 5: Well-structured, easy to extend, good error handling\n   - 3: Works but hard to modify, some technical debt\n   - 1: Monolithic, brittle, unmaintainable\n\n4. **Performance (1-5)**: Will this be fast enough?\n   - 5: Optimal, no N+1 queries, efficient algorithms\n   - 3: Acceptable, minor optimizations possible\n   - 1: Slow, will cause problems at scale\n\n5. **Security (1-5)**: Are there vulnerabilities?\n   - 5: No SQL injection, XSS, or auth issues\n   - 3: Safe but using some deprecated patterns\n   - 1: Has security vulnerabilities\n\n6. **Code Style (1-5)**: Does it follow project standards?\n   - 5: Perfect compliance (TypeScript, linting, formatting)\n   - 3: Mostly compliant, minor style issues\n   - 1: Violates standards throughout\n\n7. **Test Coverage (1-5)**: Are critical paths tested?\n   - 5: Happy path + error cases + edge cases covered, 85%+ coverage\n   - 3: Main path covered, some gaps\n   - 1: Minimal/no tests\n\n**Approval Criteria**:\n- **APPROVE** if: Correctness=5, Readability≥3, Maintainability≥3, all others ≥3\n- **REQUEST CHANGES** if: Any dimension <3 OR Correctness <5\n- **APPROVE WITH MINOR NOTES** if: 5/7 scores are 5, others ≥3\n\n### 4. Reviewer Posts Review\n\n**Review Comment Template**:\n```\n## Code Review: [TASK-XXX]\n\n### Scores\n- Correctness: 5/5 ✅\n- Readability: 4/5 ✅\n- Maintainability: 4/5 ✅\n- Performance: 5/5 ✅\n- Security: 5/5 ✅\n- Code Style: 5/5 ✅\n- Test Coverage: 4/5 ✅\n\n### Feedback\n[Line-by-line comments with suggestions, praise, or required changes]\n\n### Decision\n✅ **APPROVED** - Ready to merge\n(or) 🔄 **REQUEST CHANGES** - See comments above\n(or) ⚠️ **APPROVED WITH NOTES** - Minor suggestions for future\n```\n\n### 5. Implementer Addresses Feedback\n\nIf \"REQUEST CHANGES\":\n- [ ] Fix all requested changes\n- [ ] Re-run tests locally\n- [ ] Commit: `git commit -m \"Address review feedback on TASK-XXX\"`\n- [ ] Push: `git push`\n- [ ] @ mention Reviewer: \"Ready for re-review\"\n\nReviewer re-reviews within **2 hours**.\n\n### 6. Merge to Develop\n\nOnce APPROVED:\n- [ ] Reviewer clicks \"Approve\"\n- [ ] Implementer merges to `develop` branch\n- [ ] Implementer marks task complete in `/opt/backlog/tracking/SPRINT-001-DAILY-LOG.md`\n- [ ] Next task in queue starts\n\n---\n\n## Daily Review Window\n\n**AvailableReviewer Hours**: 09:00-17:00 UTC (8 hours per day)\n\n**PR Submission Windows** (to hit 4-hour SLA):\n\n| Submission Time | Review By | Notes |\n|---|---|---|\n| Before 13:00 | 17:00 same day | Can review same business day |\n| 13:00-17:00 | 13:00 next day | Reviewed next morning |\n| After 17:00 | 13:00 next day | Reviewed next morning |\n\n**Example Timeline**:\n- **Monday 10:00**: Implementer submits PR → Reviewed by Monday 14:00\n- **Monday 15:00**: Implementer submits PR → Reviewed by Tuesday 11:00 (next morning)\n\n---\n\n## Blocker Resolution Escalation\n\nIf PR is BLOCKED (blocking other work):\n\n1.
